Build a new endpoint (does not exist yet):

app.get("/api/proposal-log/entry/:id", async (req, res) => {
  const [entry] = await db.select().from(proposalLogEntries)

## 6. CALCULATION ENGINE

### Order of operations (EXACT — do not deviate):

```
1. Material = SUM(unit_cost × qty) for all per-item line items
2. Lump Sum Adjustment = For lump_sum quotes: MAX(0, lump_sum_total - sum of individual item costs in that quote)
3. Effective Material = Material + Lump Sum Adjustment
4. Escalation = For each item: (unit_cost × qty) × (esc_rate / 100)
   - esc_rate cascades: item.escOverride ?? category.escOverride ?? default_esc
   - Lump sum adjustment also gets escalation at the category rate
5. Freight = SUM(quote.freight) for all quotes in category (flat dollar, NOT percentage)
6. Subtotal = Effective Material + Escalation + Freight
7. Overhead (OH) = Subtotal × (oh_rate / 100)
   - oh_rate cascades: category.ohOverride ?? default_oh
   - OH changes at any level require executive approval
8. Fee = Subtotal × (fee_rate / 100)
   - fee_rate cascades: category.feeOverride ?? default_fee
   - Fee can be adjusted freely by the estimator

8. Tax = Effective Material × (tax_rate / 100)
   - Tax is on MATERIAL ONLY, not on subtotal, not on OH or Fee
9. Bond = Subtotal × (bond_rate / 100)
10. Category Total = Subtotal + OH + Fee + Tax + Bond
11. Grand Total = SUM of all category totals
```

### Override impact reporting:

When a category OH, Fee, or escalation is overridden, show the dollar impact: `Override impact: +$X vs default`

### CRITICAL: Tax on material only, not on escalation, freight, OH, or Fee.

CRITICAL: Overhead changes require executive approval. Fee is estimator-adjustable..

## 7. VENDOR QUOTE STRUCTURE

### Two pricing modes:

- **per_item**: Each line item has its own unit_cost. Quote is a container for grouping items from one vendor.
- **lump_sum**: Vendor quotes one total price for all items. Line items may have $0 unit_cost. The lump_sum_total is tracked on the quote record.

### A single category can have multiple quotes mixing both modes.

### taxIncluded flag:

- Boolean on each quote
- For ACCOUNTING VISIBILITY ONLY — so AP knows whether vendor’s price includes tax before paying
- Does NOT change the tax calculation in the estimate (tax is always calculated on material)

### Freight:

- Flat dollar amount per vendor quote (NOT a percentage)
- Stored on the quote record, not the line item

### Breakout-scoped quotes:

- breakout_group_id on quote (nullable)
- NULL = parent-level quote (applies to all breakouts)
- Set = quote is for that specific breakout group only (different pricing per building)

## 17. BREAKOUT MANAGEMENT SYSTEM

### Core concept:

Breakouts are allocation layers on the parent estimate. Line items exist ONCE in the parent. Breakout groups define how quantities are distributed.

### PV# convention: PV-2026-0147.B1, PV-2026-0147.B2 (views, not separate records)

### UI: Hidden until needed

- Small “Breakouts” button in Stage 2 header
- No breakout UI visible in default flow
- Can be added before OR after estimate is complete

### Breakout groups:

- code (B1, B2, PH1, FL3, etc.)
- label (“Building 1 - Main Tower”)
- type (building | phase | floor | scope_split | custom)
- ohp_override (nullable — inherit parent by default)
- esc_override (nullable — inherit parent by default)
- freight_allocation_method (proportional | manual)

### Allocations:

- Per line item, per breakout group: allocated_qty
- Zero allocation = item excluded from that breakout
- HARD GATE: SUM(allocations) MUST = parent qty per item
- Cannot save if violated

### Validation engine:

- Checks every item with any allocation
- Flags over-allocation (“Parent: 10, Allocated: 12, +2 over”)
- Flags under-allocation (“Parent: 10, Allocated: 8, 2 unallocated”)
- Status indicator: “35/47 items allocated”

### Bulk allocation tools:

- “All → B1” — assigns full qty of all items in current category to one group
- “Split Evenly” — distributes quantities across all groups

### Breakout-scoped quotes:

- Vendor quote can be scoped to a specific breakout group
- breakout_group_id on the quote record (nullable)
- NULL = parent-level (all breakouts), Set = that group only
- When scoped quotes have different unit costs, parent shows weighted average

### Breakout totals:

- Each group calculates: material, escalation, freight (proportional or manual), OH&P, tax, bond, total
- Reconciliation: SUM(breakout totals) should ≈ parent total
- Variance allowed only from per-breakout markup overrides — flagged and explained

### Proposal output with breakouts:

- “Pricing Breakout Summary” table after grand total showing each group’s total and item count

## 21. TAX TRACKING & REPORTING

### Tax calculation:

- Tax is calculated on MATERIAL ONLY (effective material cost)
- Tax is NOT applied to escalation, freight, overhead, or fee
- When tax_rate = 0, the proposal says “Sales tax NOT included — excluded from this proposal”
- When tax_rate > 0, the proposal says “Sales tax included (X%)”

### Vendor tax status:

- Every vendor quote has a `tax_included` boolean flag
- AI Quote Parser auto-detects tax language in quotes
- The Tax Summary card (Stage 4) shows:
  - Current tax rate and total tax amount
  - Count of vendor quotes with tax included vs excluded
  - Warning when any vendor quotes include tax (double-payment risk)
- This data flows to accounting for AP verification

### Database fields for tax reporting:

- `tax_rate` on estimates table
- `tax_included` on estimate_quotes table
- `tax_total` on estimates table (computed on save)
- `tax_status` on estimates table (“included” | “excluded” | “partial”)
- `vendor_tax_included_count` and `vendor_tax_excluded_count` on estimates table

### Tax Analytics:

- Tax inclusion rate by vendor
- Tax exposure tracking (double-payment risk)
- Tax rate by region
- Total tax collected vs tax-included vendor quotes
- Sales tax as % of bid total

## 22. OVERHEAD & FEE APPROVAL WORKFLOW

### Split markup structure:

- **Overhead (OH)**: Default 8%. Applied to subtotal. Changes require executive approval.
- **Fee**: Default 12%. Applied to subtotal. Estimator can adjust freely.

### Approval rules:

- Changing the OH default rate requires executive approval
- Changing OH per-category override requires executive approval
- Changing OH per-breakout-group override requires executive approval
- Fee has no approval gate — estimators adjust as needed for competitiveness
- All OH change requests are logged with: who requested, when, old rate, new rate, status (pending/approved/denied), who approved/denied, when

### Database:

```
TABLE: oh_approval_log
  id, estimate_id (FK)
  category (nullable — null = default rate change)
  breakout_group_id (nullable)
  old_rate, new_rate
  requested_by, requested_at
  approved_by (nullable), approved_at (nullable)
  status (enum: pending | approved | denied)
```

const userId = (req.session as any)?.userId;
