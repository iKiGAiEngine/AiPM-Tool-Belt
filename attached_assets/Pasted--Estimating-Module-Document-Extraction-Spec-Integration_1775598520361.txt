# Estimating Module — Document Extraction & Spec Integration

## OVERVIEW

Two extraction tools are built into the Estimating Module, both accessible from Stage 1 (Project Intake) as the PRIMARY location and Stage 2 (Line Items) as a secondary location:

1. **Schedule Converter** — extracts line items (types, quantities, models) from plan screenshots or pasted text. Outputs go directly into the estimate's line items with auto-scope assignment.

2. **Spec Extractor** — extracts specification sections from project specs. Outputs get tagged to the corresponding scope section and stored so the estimator can quickly reference the relevant spec language while working on each scope.

Both tools reuse the same AI logic as the existing standalone AiPM tools but output directly into the Estimating Module instead of TSV/Excel/clipboard.

### Why Stage 1

Extraction on Stage 1 answers two questions with one action:
1. **What scope sections are in this project?** — the AI auto-checks scope boxes based on what it finds
2. **What are the line items?** — items queue up and land in their scope tabs when the estimator advances to Stage 2

One upload, two outputs, zero manual data entry.

## TOOL 1: SCHEDULE CONVERTER (Line Item Extraction)

### Intake

Full-width panel opens with two intake tabs:
- **📷 Upload Images** — drag-and-drop or click to upload screenshots, photos of plans, PDFs. Supports multiple files in one batch.
- **📋 Paste Text** — textarea for pasting schedule text, spec sections, or typed quantities from emails/PDFs.

Both tabs support multiple inputs combined into one extraction.

### Processing

- Images → `POST /api/estimates/:id/extract-images` (reuses the same OCR/vision logic as `POST /api/toolbelt/schedule-to-estimate`)
- Text → `POST /api/estimates/:id/extract-text` (reuses the same parsing logic as `POST /api/toolbelt/schedule-text-to-estimate`)
- Both return `ExtractedEstimateItem[]` — the standard `ScheduleItem` structure extended with scope suggestion fields

### Extended output structure

```typescript
interface ExtractedEstimateItem {
  // Standard ScheduleItem fields
  planCallout: string;        // e.g., "10-3A"
  description: string;        // item description
  manufacturer: string;       // MFR name
  rawModel: string;           // original model text
  modelNumber: string;        // cleaned model
  quantity: number;           // numeric qty
  sourceSection: string;      // page/source reference
  confidence: number;         // 0-100 extraction confidence
  flags: string[];            // warnings
  needsReview: boolean;       // flagged for review

  // NEW — scope suggestion fields
  suggestedScope: string | null;     // scope section ID (e.g., "accessories")
  suggestedScopeCsi: string | null;  // CSI code (e.g., "10 28 00")
  scopeConfidence: number;           // 0-100 scope suggestion confidence
}
```

### Review table

The extracted items appear in a review table:

|---|---|---|---|---|---|---|---|
| ☑ | 10-3A | Grab Bar 42" | Bobrick | B-5806x42 | 24 | 95% | 🟢 Toilet Accessories ▾ |
| ☑ | 10-3B | Paper Towel Disp | Bobrick | B-2621 | 12 | 90% | 🟢 Toilet Accessories ▾ |
| ☑ | 10-5A | Partition OH Braced | — | Custom | 32 | 85% | 🟢 Toilet Partitions ▾ |

| ☐ | — | Towel warmer | — | — | 2 | 40% | 🔴 Unassigned ▾ |

All fields are inline-editable. Confidence is color-coded: green 80%+, yellow 60-79%, red <60%.

### Scope assignment dropdown logic

**If scope sections are already checked on Stage 1:**
- Dropdown shows ONLY the checked scopes + "Unassigned" + "Add scope..." option at the bottom
- AI auto-suggests within those checked scopes
- If an item doesn't match any checked scope, it shows as "Unassigned" in red
- Clicking "Add scope..." checks that scope box on Stage 1 and adds it to all dropdowns

**If no scopes are checked yet (fresh project):**
- Dropdown shows ALL Division 10 scope sections
- AI auto-suggests across the full list
- As items get assigned (auto or manual), the corresponding scope checkboxes AUTO-CHECK on Stage 1
- After review, a summary appears: "3 scope sections identified: Accessories (15 items), Partitions (6 items), Fire Protection (3 items)"

### Auto-scope suggestion keyword mapping

The AI prompt includes instructions to suggest a scope based on description keywords:

```

                        diaper changing station, baby changing

                        headrail, overhead braced, floor mounted, ceiling hung

toilet_compartments:    compartment, stall, door, latch, hinge (partition context)

lockers:                locker, storage locker, employee locker, gym locker, phenolic

fire_ext:               fire extinguisher, fire ext cabinet, FEC, fire blanket

                        directory board, poster frame

corner_guards:          corner guard, wall guard, bumper guard, chair rail

signage:                sign, signage, room sign, ADA sign, wayfinding, plaque, directory

wall_protect:           wall protection, handrail, crash rail, wall panel,
                        door protection, kick plate, push plate, pull plate

```

Items that don't match any keyword → "Unassigned" (red).
Items detected as NOT Division 10 (plumbing fixtures, electrical, mechanical) → "Not Div 10" and pre-deselected.

### Bulk actions

Above the review table:
- **"Assign All to [scope]" dropdown** — for single-scope extractions
- **Select All / Deselect All checkbox**
- **"Only show unassigned" toggle** — filter to items needing manual assignment
- **"Only show needs review" toggle** — filter to low-confidence items
- **Summary line:** "24 items extracted, 22 assigned, 2 unassigned, 3 need review"

### Send to estimate

Button: **"Send X Items to Estimate"**

Validation before send:
- All selected items MUST have a scope assignment. Block if any are "Unassigned."
- Error: "2 items have no scope assigned. Assign a scope or deselect them."

On send:
1. Create `estimate_line_items` for each selected item:
   - `name` = description
   - `model` = modelNumber
   - `mfr` = manufacturer
   - `qty` = quantity
   - `category` = assigned scope section ID
   - `csi` = CSI code from the scope section (or item-specific if more specific)
   - `unit_cost` = 0 (pricing comes from vendor quotes)
   - `source` = "extracted"
   - `has_backup` = false
   - `plan_callout` = planCallout (stored for traceability)
   - `extraction_confidence` = confidence score
   - `quote_id` = null
   - `note` = ""
2. Auto-check any scope sections on Stage 1 that received items
3. Close the extractor panel
4. Show success toast: "24 items added across 3 scope sections"
5. If on Stage 1: stay on Stage 1. Scope checkboxes now show item counts.
6. If on Stage 2: navigate to the first scope section that received items.

### Duplicate detection

If items already exist in the estimate (from a previous extraction or manual entry), check for duplicates before creating:
- Match on: same model number + same scope section
- If duplicates found, warn: "3 items may already exist in your estimate. Add anyway?"
- Options: "Add All" / "Skip Duplicates" / "Cancel"

## TOOL 2: SPEC EXTRACTOR (Specification Section Extraction & Storage)

### Purpose

The Spec Extractor pulls relevant specification language from the project specs and stores it per scope section. This gives the estimator quick reference to the spec requirements while pricing each scope — no more flipping through a 500-page spec document to find the one paragraph about toilet accessories.

### Intake

Same panel style as the Schedule Converter. Two intake tabs:
- **📷 Upload Spec Pages** — drag-and-drop spec page screenshots, PDFs, or photos
- **📋 Paste Spec Text** — textarea for pasting spec sections

### Processing

- Images → `POST /api/estimates/:id/extract-spec-images` (reuses the same AI logic as the existing Spec Extractor endpoints)
- Text → `POST /api/estimates/:id/extract-spec-text`
- The AI identifies which Division 10 specification sections are present and extracts the relevant content

### Output structure

```typescript
interface ExtractedSpecSection {
  scopeId: string;              // e.g., "accessories"
  scopeLabel: string;           // e.g., "Toilet Accessories"
  csiCode: string;              // e.g., "10 28 00"
  specSectionNumber: string;    // e.g., "10 28 00", "10 21 13.19"
  specSectionTitle: string;     // e.g., "TOILET ACCESSORIES"
  content: string;              // the actual spec language (full text)
  manufacturers: string[];      // manufacturers listed in spec
  keyRequirements: string[];    // bullet points of key requirements
  substitutionPolicy: string;   // "no substitutions" / "or equal" / "as approved"
  confidence: number;           // extraction confidence
  sourcePages: string;          // which pages this came from
}
```

### Save to estimate

Button: **"Save Spec Sections to Estimate"**

On save:
1. Create `estimate_spec_sections` records for each selected section (NEW TABLE — see database section below)
2. Auto-check the corresponding scope sections on Stage 1
3. Show success toast: "3 spec sections saved"
4. The spec sections are now accessible from within each scope tab in Stage 2

### Spec-informed RFQ generation

When the RFQ generator creates emails for a scope section, it can automatically include the spec requirements:

```

  - Grab Bar 42" (B-5806x42) — Qty: 24
  - Paper Towel Dispenser (B-2621) — Qty: 12

  - Substitution policy: Or equal (submit for approval)
```

This gives vendors spec context with the quote request — fewer clarification emails.

## NEW DATABASE TABLE: estimate_spec_sections

```
TABLE: estimate_spec_sections
  id (uuid, primary key)
  estimate_id (uuid, FK NOT NULL → estimates)
  scope_id (text — scope section ID, e.g., "accessories")
  csi_code (text — e.g., "10 28 00")
  spec_section_number (text — may be more specific, e.g., "10 28 00.13")
  spec_section_title (text — e.g., "TOILET ACCESSORIES")
  content (text — full spec text, may be long)
  manufacturers (jsonb — array of manufacturer names)
  key_requirements (jsonb — array of key requirement strings)
  substitution_policy (text — "no substitutions" / "or equal" / "as approved")
  source_pages (text — page references)
  extraction_confidence (integer — 0-100)
  created_at (timestamp)
  updated_at (timestamp)
```

Also add to `estimate_line_items`:
```
  plan_callout (text, nullable)              — from Schedule Converter extraction
  extraction_confidence (integer, nullable)  — 0-100 confidence score
  source — add "extracted" as a new valid value alongside "vendor_quote", "price_book", "library"
```

## SCOPE AUTO-CHECK BEHAVIOR (BOTH TOOLS)

Both the Schedule Converter and Spec Extractor can auto-check scope sections on Stage 1. The rules:

1. **Schedule Converter extracts items → scopes auto-check based on where items are assigned**
2. **Spec Extractor extracts spec sections → scopes auto-check based on which spec sections are found**
3. **Both can run independently or together.** Running the Spec Extractor first gives the estimator the spec context. Running the Schedule Converter gives the line items. Running both gives the complete picture.
4. **Scopes can still be manually checked/unchecked.** Auto-check adds to the selection, never removes. If the estimator unchecks a scope that has items or spec sections attached, warn: "This scope has 7 items and 1 spec section. Removing it will archive them. Continue?"
5. **The scope checkboxes show counts after extraction:**
   ```
   ☑ Toilet Accessories (10 28 00) — 15 items, 1 spec section
   ☑ Toilet Partitions (10 21 13) — 6 items, 1 spec section
   ☑ Fire Protection (10 44 00) — 3 items, 1 spec section
   ☐ Lockers (10 51 00)
   ☐ Signage (10 14 00)
   ```

## RECOMMENDED WORKFLOW (the ideal estimator experience)

1. Open project from Proposal Log → lands on Stage 1
2. Click **"Extract from Specs"** → upload spec pages → AI identifies 3 Division 10 spec sections → approve → scopes auto-check, spec language stored
3. Click **"Extract from Schedules"** → upload plan screenshots → AI extracts 24 line items, auto-assigns to the 3 scopes → review, approve → items queued
4. Advance to Stage 2 → line items already populated in each tab, spec reference panel available at top of each tab
5. Click "Send RFQ" → RFQ emails have item details AND spec requirements → send to vendors
6. Wait for quotes → enter pricing → submit bid

Total manual data entry: near zero. The estimator's job becomes review-and-approve instead of type-and-retype.

## API ENDPOINTS SUMMARY

```
# Schedule Converter (line item extraction)
POST /api/estimates/:id/extract-images      — image intake, returns ExtractedEstimateItem[]
POST /api/estimates/:id/extract-text        — text intake, returns ExtractedEstimateItem[]

# Spec Extractor (spec section extraction)
POST /api/estimates/:id/extract-spec-images — image intake, returns ExtractedSpecSection[]
POST /api/estimates/:id/extract-spec-text   — text intake, returns ExtractedSpecSection[]
POST /api/estimates/:id/save-spec-sections  — save approved spec sections to estimate

# Read spec sections (for Stage 2 reference panel)
GET  /api/estimates/:id/spec-sections       — list all saved spec sections
GET  /api/estimates/:id/spec-sections/:scopeId — get spec section for a specific scope
```

## EDGE CASES

1. **Duplicate items on second extraction:** Check for existing items with same model + scope. Warn and offer "Add All / Skip Duplicates / Cancel."
2. **Empty extraction:** "No Division 10 items found. Try a different screenshot or paste the schedule text."
3. **Not Division 10:** Items detected as plumbing/electrical/mechanical flagged "Not Div 10" and pre-deselected.
4. **Large extractions (100+ items):** Paginate review table, 25 per page. "Send All X Selected" works across pages.
5. **Spec section not matching any scope:** Show as "Unmatched" with dropdown to assign manually.
6. **Multiple spec extractions for same scope:** Second extraction appends content with divider "--- Extracted from additional pages ---" rather than overwriting.
7. **Spec section with no corresponding line items:** Valid — spec reference panel still shows even if no items priced yet.
