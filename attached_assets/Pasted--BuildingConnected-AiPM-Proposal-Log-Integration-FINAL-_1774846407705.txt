## PART 1: Autodesk OAuth Flow

### 1A: New database table — `aps_tokens`

Add to `shared/schema.ts`:

```
aps_tokens:
  id: serial PRIMARY KEY
  userId: integer NOT NULL (FK to users.id — the AiPM user who authorized)
  accessToken: text NOT NULL
  refreshToken: text NOT NULL
  expiresAt: timestamp NOT NULL
  scope: varchar(500)
  createdAt: timestamp NOT NULL default now()
  updatedAt: timestamp NOT NULL default now()
```

### 1B: New route file — `server/autodesk/auth.ts`

Create these routes:

**`GET /api/autodesk/login`** (requires auth)

- Builds the Autodesk OAuth2 authorization URL:
  - Base: `https://developer.api.autodesk.com/authentication/v2/authorize`
  - Params: `response_type=code`, `client_id` from env `APS_CLIENT_ID`, `redirect_uri=https://aipmapp.com/auth/autodesk/callback`, `scope=data:read`
- Redirects the user’s browser to that URL

**`GET /auth/autodesk/callback`** (NOTE: this path is NOT under /api — it’s the exact callback URL registered in APS. Make sure this route does NOT require auth middleware since the user is returning from an external redirect and the session needs to be read from the cookie.)

- Receives `code` query param from Autodesk
- Exchanges the code for tokens: `POST https://developer.api.autodesk.com/authentication/v2/token`
  - Body: `grant_type=authorization_code`, `code`, `redirect_uri=https://aipmapp.com/auth/autodesk/callback`, `client_id`, `client_secret`
  - Content-Type: `application/x-www-form-urlencoded`
- Stores `access_token`, `refresh_token`, `expires_at` in `aps_tokens` table linked to the session user
- Redirects to `/project-log?bc=connected` with a success indicator

**`GET /api/autodesk/status`** (requires auth)

- Returns `{ connected: true/false }` based on whether the current user has a valid (non-expired or refreshable) token in `aps_tokens`

### 1C: Token refresh helper — `server/autodesk/tokenManager.ts`

Create a helper function `getValidToken(userId)`:

- Looks up the user’s token in `aps_tokens`
- If `expiresAt` is in the future → return the access token
- If expired → call `POST https://developer.api.autodesk.com/authentication/v2/token` with `grant_type=refresh_token`, `refresh_token`, `client_id`, `client_secret`
  - Content-Type: `application/x-www-form-urlencoded`
- Update the stored token with new values
- Return the fresh access token
- If refresh fails → return null (user needs to re-authorize)

## PART 2: BC Sync Engine

### 2A: New database tables

Add to `shared/schema.ts`:

**`bc_sync_log`** — tracks every BC opportunity we’ve seen:

```
  id: serial PRIMARY KEY
  bcOpportunityId: varchar(100) NOT NULL UNIQUE
  bcProjectId: varchar(100) nullable (the shared project ID that links bid packages)
  proposalLogEntryId: integer nullable (FK to proposal_log_entries.id)
  scopeDescription: text nullable (bid package title / trade description)
  rawData: jsonb nullable (full API response for this opportunity)
  bcUpdatedAt: varchar(50) nullable (the updatedAt value from BC, for change detection)
  syncedAt: timestamp NOT NULL default now()
  updatedAt: timestamp NOT NULL default now()
```

**`bc_sync_state`** — tracks the last sync timestamp:

```
  id: serial PRIMARY KEY
  lastSyncAt: varchar(50) nullable (ISO timestamp of last successful sync)
  updatedAt: timestamp NOT NULL default now()
```

**`notifications`** — new notification system:

```
  id: serial PRIMARY KEY
  userId: integer nullable (FK to users.id — null means visible to all users)
  type: varchar(50) NOT NULL (draft_created, draft_scope_updated, draft_bc_updated, draft_approved, draft_rejected)
  title: varchar(500) NOT NULL
  message: text NOT NULL
  metadata: jsonb nullable (extra data like project name, entry ID, what changed)
  isRead: boolean NOT NULL default false
  createdAt: timestamp NOT NULL default now()
```

### 2B: New columns on `proposal_log_entries`

ADD these columns (do not remove or rename any existing columns):

```
  isDraft: boolean default false
  bcProjectId: varchar(100) nullable
  bcOpportunityIds: text nullable (JSON stringified array of BC opportunity IDs)
  scopeList: text nullable (JSON stringified array of scope descriptions, e.g. ["Toilet Accessories", "Lockers"])
  draftApprovedBy: varchar(200) nullable
  draftApprovedAt: timestamp nullable
  bcUpdateFlag: boolean default false
  bcChangeLog: text nullable (JSON stringified array of change records: [{field, oldValue, newValue, changedAt}])
```

### 2C: New route file — `server/autodesk/bcSync.ts`

Create a TWO-STEP sync flow: preview first, then confirm.

-----

**`POST /api/bc/sync/preview`** (requires admin)

This is the preview/dry-run. It runs all the logic but writes NOTHING to the database. Returns what WOULD happen.

```
Step 1: Get a valid APS token using getValidToken()
  - If no valid token → return error { success: false, error: "Not connected to BuildingConnected. Please reconnect." }

Step 2: Determine the sync time window
  - Query bc_sync_state for lastSyncAt
  - If lastSyncAt is null (FIRST SYNC): set the filter to 7 DAYS AGO only.
    Use: new Date(Date.now() - 7 * 24 * 60 * 60 * 1000).toISOString()
    This prevents a flood of historical invites on the first sync.
  - If lastSyncAt exists: use it normally to only get opportunities updated since last sync.

Step 3: Call the BC API
  - URL: GET https://developer.api.autodesk.com/construction/buildingconnected/v2/opportunities
  - Header: Authorization: Bearer {access_token}
  - Query params: filter[updatedAt] with the timestamp from Step 2
  - Handle pagination: if the response includes a "next" link or pagination cursor, follow it until all pages are fetched
  - Collect ALL opportunities into an array

Step 4: Process each opportunity through the filter gates (READ ONLY — no database writes)

  Initialize these arrays:
    toCreate = []
    toMerge = []
    toUpdate = []
    skippedCount = 0

  FOR EACH opportunity:

    --- GATE 1: GC FILTER ---
    - Look at the opportunity's client/company object for the GC name
    - The field path in the BC API response is likely: opportunity.client.name or opportunity.company.name
    - Check if the GC name exactly equals a value in the allowlist:
      const GC_ALLOWLIST = ["Swinerton Builders"];
    - If NOT on the allowlist → increment skippedCount, continue to next opportunity

    --- GATE 2: DUPLICATE CHECK ---
    - Query bc_sync_log for this opportunity's unique ID
    - IF found AND the BC updatedAt matches what we stored → already processed, increment skippedCount, continue
    - IF found AND the BC updatedAt is NEWER → this is an UPDATE, add to toUpdate array with change details
    - IF not found → this is NEW (continue to Gate 3)

    --- GATE 3: PROJECT GROUPING ---
    - Extract the BC project ID from the opportunity data
      (The field path is likely: opportunity.project.id or opportunity.projectId)
    - Query proposal_log_entries for any row where bcProjectId matches this project ID
    - Also check the toCreate array for any pending create with the same BC project ID
    - IF a match is found in DB or toCreate → this is a MERGE, add to toMerge array
    - IF no match → this is a NEW DRAFT, add to toCreate array

    --- SAFETY CAP ---
    - If toCreate.length reaches 50 → stop processing further opportunities
    - Add a warning flag: capReached = true

Step 5: Return the preview response (NO database writes):
  {
    success: true,
    preview: {
      toCreate: [
        { projectName, gcName, dueDate, scope, location, bcOpportunityId, bcProjectId, rawData }
      ],
      toMerge: [
        { projectName, existingScope: [...], newScope, bcOpportunityId, existingEntryId }
      ],
      toUpdate: [
        { projectName, changes: [{field, from, to}], bcOpportunityId, existingEntryId }
      ]
    },
    summary: {
      toCreate: toCreate.length,
      toMerge: toMerge.length,
      toUpdate: toUpdate.length,
      skipped: skippedCount,
      capReached: capReached || false,
      capWarning: capReached ? "Reached maximum of 50 new drafts. Run sync again for remaining." : null
    }
  }
```

**`POST /api/bc/sync/confirm`** (requires admin)

This actually writes to the database. It re-fetches and re-processes (or accepts the preview data) to ensure consistency.

```
Step 1: Re-run the same fetch and filter logic from the preview (or re-use cached preview if within a short time window)

Step 2: For each item in toCreate — CREATE NEW DRAFT:
  - Create a new proposal_log_entries row:
    - projectName: from BC opportunity (likely opportunity.name or opportunity.project.name)
    - gcEstimateLead: from BC opportunity client contact name
    - dueDate: from BC opportunity due date (format to match existing format in the table — inspect what format existing rows use, likely MM/DD/YY or YYYY-MM-DD)
    - region: parse from BC project location/address using this mapping:
        "Colorado" or "Denver" → "Colorado (DEN)"
        "Atlanta" or "Georgia" → "Atlanta (ATL)"
        "San Francisco" or "Sacramento" or "Northern Cal" → "Nor Cal (SFO)"
        "Los Angeles" or "Southern Cal" or "Orange County" → "SPD (LAX)"
        "Seattle" or "Washington" → "Washington (SEA)"
        "Portland" or "Oregon" → "Portland (PDX)"
        "Austin" or "Texas" → "Austin (AUS)"
        If no match → leave null for manual assignment
    - estimateStatus: "Draft" (ADD "Draft" as a new valid status in the system)
    - isDraft: true
    - inviteDate: BC opportunity created/received timestamp (formatted to match existing date format)
    - bcProjectId: the BC project ID
    - bcOpportunityIds: JSON.stringify([this opportunity ID])
    - scopeList: JSON.stringify([bid package title or trade description from this opportunity])
    - bcLink: the BC opportunity URL if available in the API response
    - notes: "Auto-created from BuildingConnected sync"
    - DO NOT set estimateNumber yet — that gets assigned when the draft is approved
  - Insert a bc_sync_log record linking this opportunity ID to the new proposal log entry
  - Create a notification (type: "draft_created"):
    Title: "New BC Invite: [projectName]"
    Message: "Swinerton Builders — [projectName]. Due: [dueDate]. Scope: [scope]. Review in Proposal Log."
    userId: null (visible to all users)
  - Send a SendGrid email to Haley (follow the same pattern used for bid assignment emails):
    Subject: "AiPM: New Bid Invite — [projectName]"
    Body: same info as notification + link to Proposal Log

Step 3: For each item in toMerge — MERGE SCOPE:
  - Find the existing proposal_log_entries row by bcProjectId
  - Parse the existing scopeList JSON array
  - Append the new bid package title/trade description
  - Update scopeList with the new JSON array
  - Parse the existing bcOpportunityIds JSON array
  - Append this opportunity ID
  - Update bcOpportunityIds
  - If the new invite has an EARLIER due date than stored → update dueDate
  - Insert a bc_sync_log record linking this opportunity to the existing entry
  - Create a notification (type: "draft_scope_updated"):
    Title: "Scope Updated: [projectName]"
    Message: "New scope added: [new scope]. Full scope: [all scopes comma-separated]."
    userId: null
  - If nbsEstimator is set on the row, also create a notification for that estimator's user ID
  - Send a SendGrid email to Haley

Step 4: For each item in toUpdate — UPDATE EXISTING:
  - Find the bc_sync_log record for this opportunity
  - Compare new data with stored rawData to identify changes
  - Key fields to check: due date, project name, location, scope description
  - For each changed field:
    - Update the corresponding proposal_log_entries column
    - Parse existing bcChangeLog (or initialize as empty array)
    - Append: {field, oldValue, newValue, changedAt: new Date().toISOString()}
    - Update bcChangeLog
  - Set bcUpdateFlag = true
  - Update bc_sync_log with new rawData and bcUpdatedAt
  - Create a notification (type: "draft_bc_updated"):
    Title: "Invite Updated: [projectName]"
    Message: "[field] changed from [old] to [new] by Swinerton."
    userId: null
  - If nbsEstimator is set, also notify that estimator
  - Send a SendGrid email to Haley

Step 5: Update bc_sync_state with current timestamp as lastSyncAt

Step 6: Return summary:
  { success: true, created: X, merged: X, updated: X, skipped: X }
```
